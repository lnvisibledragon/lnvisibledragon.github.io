시스템적인 측면에서는 pub/sub이 더 부담없어 보임. 다만, 수신자가 받았는지 아닌지 알 수 없음.(구독이 끊겼을 시 메세지 유실)

사용자 경험의 측면에서 순서보장이 확실하게 되면 좋겠다는 생각에 Redis List를 선택

이후 채팅 서버 분리를 고려해보았음. 분리를 위한 코드는 짰지만, 실질적으로 얻는 이득이 적다고 생각해 파기

이미지 저장 방식 설계

사용자들은 활동에 대한 보상으로 포인트를 얻고, 포인트로 장착아이템을 얻을 수 있음. 4개의 파츠: 모자,얼굴, 헤어,상의

초기 구상: 각각의 파츠에 대한 이미지는 프론트측에서 관리, 파츠들이 조합된 이미지는 db의 멤버 정보에 저장하고 변경 시, 멤버 테이블의 이미지url만 바뀌는 방식

초기 구상의 문제점: 멤버테이블에만 조합된 이미지가 있고, 어떤 조합인지에 대한 정보가 없음.- 멤버가 착용한 아이템이 바뀔때마다 멤버의 이미지가 바뀜. 즉, 같은 이미지였더라도 다른 url이 계속해서 생성. 이미지는 비정형이라 조합을 명시하지 않으면 어떤 이미지인지 알 수 있는 방법X

변경 후 : 각각 파츠의 아이디와 이미지를 저장하는 별도의 테이블 추가, 이미지를 변경할 때 마다 조합 테이블에 있는 조합인지 판별해서 이미지를 멤버 테이블에 삽입

이득: 이미지 url이 바뀔때 마다 새로 생성되지 않음. 손해: 어떤 조합인지 판별하는 과정이 리소스를 많이 잡아먹음.

solving: unique인덱스를 걸어 모자,헤어,얼굴,상의의 순서로 된 경우에 한해 조회 성능을 개선

단점: 파츠가 늘면 DB 자체도 수정될 필요가 있음
